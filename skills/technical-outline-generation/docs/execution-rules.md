# 技术卷大纲推导规则

## 1. 技术卷生成目标

技术卷大纲不是对招标文件原目录的简单抽取，而是根据评分标准、技术要求、采购需求、服务要求反向推导出的写作目录。

目标是：

1. 最大化覆盖技术评分项；
2. 准确响应采购需求；
3. 支撑后续正文生成；
4. 保证章节结构符合投标文件阅读习惯；
5. 避免商务、资格、报价内容混入技术卷。

## 2. 证据优先级

生成技术卷大纲时，必须按照以下优先级使用证据：

```text
评分标准 > 技术/采购需求 > 服务/实施/交付/验收要求 > 格式/目录参考
```

具体要求：

1. 评分标准是最高优先级；
2. 高分评分项必须在一级或二级章节中体现；
3. 技术/采购需求用于确定章节内容边界；
4. 实施、部署、测试、验收、运维、培训、售后要求用于补充过程类章节；
5. 格式/目录参考只用于判断技术卷格式要求；
6. 不得因为格式参考中出现商务材料，就把商务材料写入技术卷。

## 3. 评分标准解析规则

必须从 `scoringItems` 和 `scoringChunks` 中识别以下信息：

1. 评分项名称；
2. 评分项分值；
3. 评分项权重；
4. 得分要求；
5. 扣分要求；
6. 关键词；
7. 是否为高分项；
8. 是否需要独立成章；
9. 是否可与其他评分项合并表达。

## 4. 高分项判断规则

满足以下任一条件，可视为高优先级评分项：

1. `maxScore` 明显高于其他技术评分项；
2. `scoreWeight` 较高；
3. `criteriaText` 中出现“完整性”“合理性”“可行性”“先进性”“针对性”“详细”“优良”等评价词；
4. 评分项要求提供专项方案；
5. 评分项直接影响技术得分；
6. 评分项在评分标准中重复出现或被重点描述。

高分项处理要求：

1. 不得遗漏；
2. 不得只放在三级章节；
3. 优先作为一级章节或二级章节；
4. 必须在 `coverageMatrix` 中标明 `coveredBy`；
5. 如果无法覆盖，必须写入 `warnings`。

## 5. 技术/采购需求解析规则

必须从 `technicalRequirementItems` 和 `technicalRequirementChunks` 中识别：

1. 建设目标；
2. 建设内容；
3. 功能模块；
4. 技术参数；
5. 服务范围；
6. 实施要求；
7. 部署要求；
8. 测试要求；
9. 验收要求；
10. 培训要求；
11. 运维要求；
12. 售后要求；
13. 安全要求；
14. 性能要求；
15. 交付成果。

采购需求用于限定章节写作边界，不得脱离采购需求生成泛化内容。

## 6. 章节生成规则

### 6.1 一级章节

一级章节应满足：

1. 覆盖核心评分项；
2. 体现技术卷整体逻辑；
3. 使用中文序号；
4. 标题格式示例：`一、项目理解与需求分析`；
5. 不得使用过于空泛的标题；
6. 一级章节建议 5 到 12 个。

### 6.2 二级章节

二级章节应满足：

1. 对一级章节进行拆解；
2. 尽量对应具体评分点或需求点；
3. 使用数字编号；
4. 标题格式示例：`1.1 建设目标理解`；
5. 每个一级章节建议 2 到 6 个二级章节。

### 6.3 三级章节

三级章节应满足：

1. 用于支撑正文生成；
2. 不得过度碎片化；
3. 使用数字编号；
4. 标题格式示例：`1.1.1 建设背景分析`；
5. 每个二级章节建议 1 到 4 个三级章节。

## 7. chapterDepth 控制规则

必须严格遵守 `generationConfig.chapterDepth`：

1. `chapterDepth = 1`：只输出一级章节；
2. `chapterDepth = 2`：输出一级、二级章节；
3. `chapterDepth = 3`：输出一级、二级、三级章节；
4. 不得超过配置要求的章节深度。

## 8. 评分项覆盖矩阵规则

必须输出 `coverageMatrix`。

`coverageMatrix` 用于说明每个评分项被哪些章节覆盖。

每个评分项应包含：

1. `scoringItem`；
2. `maxScore`；
3. `coveredBy`；
4. `coverageStatus`；
5. `reason`。

`coverageStatus` 只能是：

1. `covered`；
2. `partially_covered`；
3. `not_covered`。

高分项原则上不得为 `not_covered`。

如果高分项为 `not_covered`，必须在 `warnings` 中说明原因。

## 9. 风险提醒规则

以下情况必须写入 `warnings`：

1. 评分标准为空；
2. 技术/采购需求为空；
3. 某些高分评分项无法覆盖；
4. 证据不足以支撑完整技术卷；
5. 招标文件中技术卷、商务卷、资格卷边界不清；
6. 格式参考中存在可能误导技术卷内容的商务材料；
7. 章节结构只能部分覆盖评分标准；
8. 大纲生成依赖 `inferred` 推导较多。

## 10. 失败兜底规则

### 10.1 评分标准为空，技术/采购需求不为空

可以基于技术/采购需求生成大纲，但必须：

1. `coverageMatrix` 输出空数组；
2. `warnings` 说明评分标准缺失；
3. 章节 `basis` 中说明依据来自技术/采购需求。

### 10.2 评分标准和技术/采购需求均为空

不得编造大纲。

必须输出：

```json
{
  "documentTitle": "投标项目技术卷大纲",
  "structureStrategy": {
    "type": "unknown",
    "reason": "缺少评分标准和技术/采购需求证据，无法判断技术卷结构策略。"
  },
  "chapters": [],
  "coverageMatrix": [],
  "warnings": [
    "缺少评分标准和技术/采购需求证据，无法可靠生成技术卷大纲。"
  ]
}
```
