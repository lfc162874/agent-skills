# 技术卷结构策略

## 1. 总体说明

技术卷大纲不能机械地把评分项逐条变成章节。

应根据评分标准、采购需求、项目类型选择合适的大纲结构。

核心原则：

```text
评分标准决定章节重点；
采购需求决定章节边界；
结构策略决定表达顺序；
覆盖矩阵证明评分项覆盖。
```

## 2. 可选结构策略

`structureStrategy` 支持以下类型：

1. `score_driven`；
2. `total_sub_total`；
3. `process_driven`；
4. `module_driven`；
5. `hybrid`；
6. `unknown`。

如果输入 `generationConfig.structureStrategy = auto`，应自动判断最合适的结构策略。

## 3. score_driven：评分项驱动结构

### 适用场景

1. 评分标准非常清晰；
2. 每个评分项都有明确分值；
3. 每个评分项适合独立成章；
4. 技术卷格式没有强制目录；
5. 项目内容相对标准化。

### 典型结构

```text
一、项目理解与需求分析
二、技术方案
三、实施方案
四、质量保障方案
五、项目团队配置
六、售后服务方案
七、应急保障方案
```

### 使用要求

1. 章节应直接对应评分项；
2. 高分评分项优先独立成章；
3. 普通评分项可合并到相关章节；
4. 必须输出 `coverageMatrix` 证明覆盖关系。

## 4. total_sub_total：总分总结构

### 适用场景

1. 综合性技术服务项目；
2. 需要体现完整方案逻辑；
3. 评分项分散但整体响应很重要；
4. 需要先总述，再分项展开，最后形成保障或承诺。

### 典型结构

```text
一、项目理解与总体响应
二、总体技术方案
三、专项技术方案
四、实施组织与进度计划
五、质量、安全与风险保障
六、运维服务与售后保障
七、总结与服务承诺
```

### 使用要求

1. 第一部分应体现总体理解和响应；
2. 中间章节覆盖主要评分项；
3. 后部章节覆盖保障、服务、风险、承诺等内容；
4. 不得为了总分总结构生成无依据的总结章节。

## 5. process_driven：流程型结构

### 适用场景

1. 软件开发项目；
2. 系统建设项目；
3. 平台交付项目；
4. 集成部署项目；
5. 数据治理项目；
6. 智能体、AI 平台、大模型应用建设项目。

### 典型结构

```text
一、项目理解与需求分析
二、总体架构设计
三、功能建设方案
四、系统集成与部署方案
五、项目实施计划
六、测试与验收方案
七、运维与服务保障
八、安全与质量保障
```

### 使用要求

1. 按项目生命周期组织章节；
2. 适合需求、设计、开发、部署、测试、验收、运维链路；
3. 如果评分标准包含实施、测试、验收、运维，应优先采用该结构或 `hybrid`；
4. 不得忽略评分项覆盖。

## 6. module_driven：模块响应型结构

### 适用场景

1. 采购需求中明确列出多个功能模块；
2. 模块边界清晰；
3. 每个模块都需要单独响应；
4. 评分标准关注功能完整性、模块设计、响应程度。

### 典型结构

```text
一、项目总体理解与建设思路
二、模型中心建设方案
三、知识库中心建设方案
四、权限中心建设方案
五、智能审查中心建设方案
六、文档导出中心建设方案
七、模块集成与协同方案
八、实施交付与服务保障
```

### 使用要求

1. 模块名称应来自采购需求，不得自造；
2. 模块章节应覆盖核心功能需求；
3. 应增加模块间集成、数据流转或协同方案；
4. 应通过 `coverageMatrix` 说明模块章节覆盖哪些评分项。

## 7. hybrid：混合结构

### 适用场景

1. 真实招标文件中评分标准和采购需求都较复杂；
2. 单一结构无法完整覆盖；
3. 既要覆盖评分项，又要保持技术卷逻辑完整；
4. 既有模块需求，又有实施、质量、服务等评分项。

### 推荐结构逻辑

```text
前部：项目理解、需求分析、总体响应；
中部：按评分项、流程或模块展开核心方案；
后部：实施、质量、安全、运维、售后、风险等保障章节。
```

### 使用要求

1. 默认优先考虑 `hybrid`；
2. 高分评分项优先体现在一级或二级章节；
3. 采购需求中的模块可作为中部章节；
4. 实施、质量、安全、运维等要求可作为后部保障章节；
5. 必须在 `structureStrategy.reason` 中说明采用 `hybrid` 的原因。

## 8. unknown

当评分标准和技术/采购需求均不足以判断结构策略时，使用 `unknown`。

使用 `unknown` 时：

1. `chapters` 应为空或极简；
2. `coverageMatrix` 应为空；
3. `warnings` 必须说明证据不足；
4. 不得编造完整技术卷大纲。
