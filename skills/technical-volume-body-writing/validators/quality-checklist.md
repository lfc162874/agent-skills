# 质量检查清单

本文件用于技术卷正文生成后的自检、后处理校验和质量审查。

## JSON 合规性

必须检查：

1. 输出是合法 JSON；
2. 根对象包含 `sectionId`、`sectionTitle`、`blocks`、`scoreCoverage`、`evidenceRefs`、`qualityHints`；
3. 不存在 schema 之外的字段；
4. `blocks` 不为空；
5. block 类型只包含 `paragraph`、`list`、`table`，仅在允许小标题时使用 `heading`；
6. 每个 block 都有合法 `styleKey`。

## 格式合规性

必须检查：

1. 不包含 Markdown 加粗符号；
2. 不包含 Markdown 标题符号；
3. 不包含 Markdown 列表符号；
4. 不包含 Markdown 表格和代码块；
5. 不包含手写章节编号；
6. 不包含手写列表编号；
7. 不包含段首空格、全角空格或制表符；
8. paragraph 中不得通过换行模拟多个段落。

## 大纲合规性

必须检查：

1. 内容属于当前章节；
2. 不生成当前章节之外的标题；
3. 不混入商务卷、资格卷、报价文件内容；
4. 不重复生成当前章节标题作为正文；
5. 未经允许不得生成小标题；
6. 小标题不得超出 `outlineTree` 或 `allowSubHeadings` 允许范围。

## 评分点覆盖

必须检查：

1. `scoreCoverage` 中列出当前章节关联评分项；
2. 高分评分项有明确响应；
3. `coverageDescription` 不是空话；
4. 正文内容能支撑 `covered=true`；
5. 如果无法覆盖评分项，应如实标记 `covered=false`，不得虚假覆盖。

## 招标要求响应

必须检查：

1. 正文回应招标文件中的技术要求；
2. 正文回应采购需求或服务要求；
3. 明确需求边界、实施措施、管理机制或交付成果；
4. 不脱离招标人项目背景泛泛写行业内容。

## 证据和事实风险

必须检查：

1. 不编造企业资质；
2. 不编造证书编号；
3. 不编造项目案例；
4. 不编造客户名称；
5. 不编造人员姓名和团队人数；
6. 不编造中标金额、合同金额、日期；
7. 使用知识库事实时，`evidenceRefs` 有对应来源；
8. 没有证据时使用通用能力表达。

## 内容质量

必须检查：

1. 语言正式、专业、稳健；
2. 内容不空泛；
3. 有具体措施、流程、机制、成果或控制点；
4. 每段主题明确；
5. 不重复堆叠同义句；
6. 不过度使用“我公司将”“本项目将”；
7. 不出现“以下是正文”“根据以上内容”等提示语；
8. 不暴露 AI 生成痕迹。

## 质量判定建议

可按以下规则处理：

1. JSON 不合法：必须重新生成；
2. 出现 Markdown 或手写编号：优先后处理清洗，严重时重新生成；
3. 大纲越界：必须重新生成；
4. 出现虚构事实风险：必须删除或改为通用表达；
5. 评分点覆盖不足：进入补强生成；
6. 内容空泛：进入正文优化生成。
